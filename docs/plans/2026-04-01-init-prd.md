# PRD: jig init ‚Äî First-Run Setup Experience

**Work type:** Feature
**Tier:** Full PRD
**Date:** 2026-04-01
**Spec reference:** [docs/init-experience.md](../init-experience.md)

---

## 1. Overview

A workflow skill (`/jig:init`) that guides teams through first-run setup
after installing Jig as a plugin. The AI auto-detects the project
environment (git host, main branch, platform, commit conventions), then
conducts a back-and-forth interview ‚Äî one question at a time ‚Äî to capture
team configuration. It generates `jig.config.md`, scaffolds the `team/`
extension directory, and modifies the project's CLAUDE.md with a Jig
declaration.

---

## 2. Background & Motivation

Teams install Jig via marketplace plugin (adding to `.claude/settings.json`),
but after install there's no guided setup. Today, teams must:

- Manually create `jig.config.md` by copying the scaffold template
- Create the `team/` directory structure by hand
- Know to add a Jig declaration to their CLAUDE.md
- Understand what each config field means and what values to use

This creates friction at the exact moment when first impressions matter.
`/jig:init` turns a manual copy-and-edit process into a 2-minute guided
conversation that produces a correct, tailored configuration.

## 3. Skill Interface

> Adapted from "API Contract" ‚Äî this is a skill, not an API.

### Input

| Source | What | How |
|--------|------|-----|
| Project environment | Git host, main branch, platform, commit convention, existing hooks, existing skills | Auto-detected via bash commands and file checks |
| User interview | Team name, ticket system, ticket prefix, engineering concerns, starter pack preference | Asked one at a time via conversation |

### Output

| Artifact | Location | Condition |
|----------|----------|-----------|
| `jig.config.md` | Project root | Always |
| `team/skills/.gitkeep` | Project root | Always |
| `team/specialists/.gitkeep` | Project root | Always |
| `team/agents/.gitkeep` | Project root | Always |
| `team/README.md` | Project root | Always |
| CLAUDE.md modification | Project root | Always (append if exists, create if not) |

### Invocation

```
/jig:init
```

Single entry point. No arguments ‚Äî everything is detected or asked.

### Errors / Abort Conditions

| Condition | Behavior |
|-----------|----------|
| `jig.config.md` already exists | Warn user. Ask: overwrite, edit, or abort. |
| Not in a git repository | Warn but proceed ‚Äî skip git-host and branch detection |
| No git remote | Skip git-host auto-detection, ask manually in Q2 follow-up |
| User cancels mid-interview | Stop gracefully, do not write partial files |

## 5. Business Logic & Rules

### Auto-Detection (runs before any questions)

All detection runs silently. Results are presented to the user in a
summary block before the first question.

#### Git Host Detection

```bash
git remote get-url origin
```

Parse the URL:
- Contains `github.com` ‚Üí `github`
- Contains `gitlab.com` or `gitlab.` ‚Üí `gitlab`
- Contains `bitbucket.org` or `bitbucket.` ‚Üí `bitbucket`
- No remote or unrecognized ‚Üí flag for manual input

#### Main Branch Detection

```bash
git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null
```

Extract branch name (e.g., `refs/remotes/origin/main` ‚Üí `main`).
Fallback: check if `main` or `master` branch exists locally.

#### Platform Detection

Check project root for:
- `CLAUDE.md` exists ‚Üí `claude`
- `GEMINI.md` exists ‚Üí `gemini`
- `AGENTS.md` exists ‚Üí `codex`
- Multiple exist ‚Üí list all detected, ask which is primary
- None exist ‚Üí default `claude` (since Jig is a Claude Code plugin)

#### Commit Convention Detection

```bash
git log --oneline -20
```

Analyze commit messages:
- Matches `type(scope): message` or `type: message` ‚Üí `conventional`
- Matches `[TYPE]` or similar ‚Üí `angular`
- No clear pattern ‚Üí default `conventional`

#### Existing Hooks Detection

Check for:
- `.husky/` directory ‚Üí husky detected
- `.commitlintrc*` or `commitlint.config.*` ‚Üí commitlint detected
- `.cz-config*` or `.czrc` ‚Üí commitizen detected

If commitlint config found: parse `type-enum` and `scope-enum` rules,
import into `jig.config.md` as `types` and `scopes`.

#### Existing Skills Detection

Check for `.claude/skills/` directory with `SKILL.md` files. If found:
- Count and list discovered skills
- Suggest migration to `team/skills/` during the summary

### Interview Flow

Questions are asked **one at a time**. Each question waits for the
user's response before proceeding to the next. The AI must parse natural
language answers and map to config values.

#### Q1: Team Name

```
What's your team name?
```

- Free text input
- Used as the `name` field in jig.config.md
- No validation ‚Äî accept whatever the user provides

#### Q2: Ticket System

```
Where do you track tickets?

  (a) GitHub Issues
  (b) Linear
  (c) Jira
  (d) Other
  (e) None ‚Äî we don't use tickets
```

- Map to `ticket-system`: `github`, `linear`, `jira`, `other`, `none`
- If user says "Linear" or "we use linear" ‚Üí map to `linear`
- If `other` ‚Üí ask what system, store as-is
- If `none` ‚Üí set `ticket-system: none`, skip Q3

#### Q3: Ticket Prefix (conditional)

```
Ticket prefix? (e.g., ENG, PROJ ‚Äî appears in branch names and PR
references)
```

- **Skip if** Q2 answer was `github` or `none` (GitHub Issues uses
  numbers, no prefix needed; no tickets means no prefix)
- Free text input
- Used as `ticket-prefix` in jig.config.md
- Also used to derive branching format (see Branching Format section)

#### Q4: Engineering Concerns

```
What engineering concerns matter to your team?
These surface during brainstorming to make sure nothing gets missed.
You can always add more later in jig.config.md.

  [ ] i18n / translations
  [ ] Analytics / event tracking
  [ ] Feature flags
  [ ] Database migrations
  [ ] Caching
  [ ] Webhooks / external notifications
  [ ] Event publishing (NATS, Kafka, etc.)
  [ ] Security / auth
  [ ] Responsive layout
```

- Multi-select: user can pick any combination
- Present as a list ‚Äî user responds naturally ("i18n, analytics, and
  caching" or "the first three" or "all of them")
- Each selected concern ‚Üí a line in the Concerns Checklist pointing
  to `manual`
- Core specialists (error-handling, security) are always included
  regardless of selection
- `test-strategy: manual` is always included

#### Q5: Engineering Starter Pack

```
Install the engineering starter pack?
Includes: copywriting standards, logging guidance, test strategy

  (y) Yes ‚Äî recommended
  (n) No ‚Äî I'll add my own
```

- If yes: note in the summary that pack skills are available via the
  plugin (they ship with Jig ‚Äî no file copying needed)
- If no: omit pack references from the summary

### Branching Format Generation

Derived from ticket system + prefix:

| Ticket System | Branch Format |
|--------------|---------------|
| GitHub Issues | `{username}/gh-{number}-{kebab-title}` |
| Linear | `{username}/{prefix}-{number}-{kebab-title}` |
| Jira | `{username}/{prefix}-{number}-{kebab-title}` |
| Other | `{username}/{prefix}-{number}-{kebab-title}` |
| None | `{username}/{kebab-title}` |

The `{prefix}` is lowercased from the ticket prefix answer.

### Config Generation

After all questions are answered, generate `jig.config.md` by populating
the scaffold template with:

1. Auto-detected values (git-host, main-branch, platform, convention)
2. Interview answers (name, ticket-system, ticket-prefix, concerns)
3. Derived values (branching format)
4. Imported values (commitlint types/scopes, if detected)
5. Default values for everything else (pipeline stages, review config,
   execution config, commit config)

## 6. Entry Points & User Flows

> Adapted from "Entry Points & User Flows" ‚Äî single entry point, no
> navigation paths.

### Primary Flow

User installs Jig plugin ‚Üí types `/jig:init` ‚Üí skill runs.

### Typical Session

1. AI runs auto-detection (2-3 seconds of bash commands)
2. AI presents detection summary
3. AI asks Q1 ‚Üí user answers ‚Üí AI acknowledges
4. AI asks Q2 ‚Üí user answers ‚Üí AI acknowledges
5. AI asks Q3 (if applicable) ‚Üí user answers ‚Üí AI acknowledges
6. AI asks Q4 ‚Üí user answers ‚Üí AI acknowledges
7. AI asks Q5 ‚Üí user answers ‚Üí AI acknowledges
8. AI generates all files
9. AI shows summary of what was created
10. AI shows next steps

### Re-Init Flow

User types `/jig:init` in a project that already has `jig.config.md`:

1. AI detects existing config
2. AI warns: "Found existing jig.config.md. Want to overwrite it with
   fresh config, or edit the existing one?"
3. If overwrite ‚Üí proceed with normal flow
4. If edit ‚Üí suggest using direct edit instead of init
5. If abort ‚Üí stop gracefully

## 9. Settings & Configuration

> Documents every field generated in jig.config.md, what populates it,
> and what it controls.

### Team Section

| Field | Source | Controls |
|-------|--------|----------|
| `name` | Q1 | Team identifier in generated output |
| `platform` | Auto-detect | Which adapter to use |
| `git-host` | Auto-detect | Git command mapping (push, PR, etc.) |
| `ticket-system` | Q2 | Ticket creation routing |
| `ticket-prefix` | Q3 | Branch naming, ticket references |

### Pipeline Section

Default values ‚Äî not customized during init:

```yaml
stages: [discover, brainstorm, plan, execute, review, ship, learn]
```

Stage overrides use the standard defaults from the scaffold template.

### Branching Section

| Field | Source | Controls |
|-------|--------|----------|
| `format` | Derived from Q2 + Q3 | Branch name pattern |
| `main-branch` | Auto-detect | Base branch for PRs |

### Concerns Checklist

Built from Q4 selections + core defaults:

- Each Q4 selection ‚Üí `- {concern}: manual`
- Always included: `error-handling: core/specialists/error-handling`
- Always included: `security: core/specialists/security`
- Always included: `test-strategy: manual`

### Review, Execution, Commit Sections

Default values from the scaffold template. Not customized during init.

## 11. Out of Scope

- **Multi-platform init** (`--platform claude,gemini`) ‚Äî documented in
  the adapter but not implemented in this iteration. Single-platform
  only (claude).
- **Pack browsing/installation** ‚Äî Q5 asks about the engineering starter
  pack, but there's no pack marketplace. The pack ships with Jig.
- **Config migration** ‚Äî upgrading from an older jig.config.md format.
  Not needed yet (no prior versions exist).
- **Interactive config editor** ‚Äî init creates the config. Editing it
  later is a direct file edit, not a re-run of init.
- **CLI mode** ‚Äî this runs inside Claude Code as a skill, not as a
  standalone CLI binary.
